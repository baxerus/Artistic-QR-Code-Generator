Modify index.html to add the QR corruption and decoding pipeline. This is a single task because the three pieces (masking, overlay, decode) form one tightly-coupled pipeline that must be tested together.

**Step 1: Inline jsQR library**
- Fetch jsQR v1.4.0 from CDN: `https://cdn.jsdelivr.net/npm/jsqr@1.4.0/dist/jsQR.js`
- Inline the minified source into a new `<script>` tag AFTER the qrcodejs script and BEFORE the application logic script
- Verify the global `jsQR` function is available after inlining

**Step 2: Switch QR generation from SVG to Canvas mode**
- The current qrcodejs renders as SVG by default (it detects SVG support and prefers it)
- For pixel-level manipulation, we need Canvas rendering
- Modify the QR generation to force canvas mode. The simplest approach: after generating the QR via `new QRCode(...)`, extract the module data from the QRCode internal model (`qrCodeInstance._oQRCode`) to get the boolean module grid, then render it ourselves onto a Canvas element
- Alternative approach: The qrcodejs library has a Canvas renderer built in. We can force it by temporarily overriding the SVG detection. Look at the library code -- it checks `"svg" === document.documentElement.tagName.toLowerCase()` and `typeof CanvasRenderingContext2D`. We can ensure the Canvas path is taken.
- **Recommended approach:** After QR generation, extract the module grid from `qrCodeInstance._oQRCode` using `isDark(row, col)` and `getModuleCount()`. Create our own canvas element, render the QR modules onto it, and replace the SVG in the container. This gives us full control over the canvas for subsequent overlay and decode operations. Store this canvas reference as `qrCanvas` globally.

Specifically, create a `renderQRToCanvas(qrModel, container)` function that:
  - Gets moduleCount from `qrModel.getModuleCount()`
  - Creates a canvas element sized appropriately (moduleCount * cellPixels where cellPixels gives ~400px total, e.g., `Math.floor(400 / moduleCount)` then `moduleCount * cellPixels` for exact canvas size)
  - Renders each module as a filled rectangle (black for dark, white for light)
  - Replaces container content with this canvas
  - Returns the canvas element
  - Uses `ctx.imageSmoothingEnabled = false` for crisp rendering

Update `generateQR()` to:
  - Generate QR using existing QRCode constructor (keep for the internal model)
  - Then call `renderQRToCanvas()` to create our own canvas
  - Store result in global `qrCanvas` variable
  - After rendering, call the overlay and decode pipeline

**Step 3: Implement function pattern masking**
Create `createFunctionPatternMask(version)` that returns a flat boolean array (size*size) where `true` means "this cell is a function pattern -- do not corrupt."

Function patterns to mask (per QR spec):
- **Finder patterns**: 7x7 at top-left (0,0), top-right (size-7, 0), bottom-left (0, size-7). Include 1-module separator around each finder, making the reserved area 8x8 or 9x9 as appropriate:
  - Top-left: rows 0-8, cols 0-8 (9x9 block)
  - Top-right: rows 0-8, cols (size-8) to (size-1) (9x8 block)
  - Bottom-left: rows (size-8) to (size-1), cols 0-8 (8x9 block)
- **Timing patterns**: All cells in row 6 and column 6 (between the finder separators, i.e., from col 8 to size-9 on row 6, and row 8 to size-9 on col 6) -- but mark the entire row 6 and column 6 to be safe
- **Alignment patterns**: 5x5 centered at coordinates from the alignment table. Use the table from RESEARCH.md:
  ```
  Version 2: [6, 18]
  Version 3: [6, 22]
  Version 4: [6, 26]
  Version 5: [6, 30]
  Version 6: [6, 34]
  Version 7: [6, 22, 38]
  Version 8: [6, 24, 42]
  ```
  For each pair of coordinates (row, col) from the cartesian product, skip if it overlaps a finder pattern area (top-left 9x9, top-right or bottom-left 8x9/9x8). Mark 5x5 centered at (row, col), i.e., (row-2, col-2) to (row+2, col+2).
- **Dark module**: Position (4*version + 9, 8) -- always dark. Mark as function pattern.
- **Format information**: The strips around the finder separators that hold format data. Already covered by the 9x9/9x8/8x9 blocks above.

**Step 4: Implement corruption overlay**
Create `overlayPaintedPixels(canvas, paintGrid, functionMask)` that:
  - Gets canvas 2D context and reads ImageData
  - For each cell in paintGrid where state !== 0 (locked white or locked black):
    - Check `functionMask[y * gridSize + x]` -- if true, SKIP (do not corrupt function patterns)
    - Calculate pixel coordinates on the canvas for this grid cell
    - Overwrite all pixels in that cell's area with white (255,255,255) for state 1 or black (0,0,0) for state 2
  - Puts the modified ImageData back onto the canvas
  - The cell size on the QR canvas must match the QR module size (which we control since we render the QR ourselves)

**Step 5: Implement QR decoding**
Create `decodeCorruptedQR(canvas)` that:
  - Gets ImageData from the canvas
  - Calls `jsQR(imageData.data, imageData.width, imageData.height)`
  - Returns `{ success: boolean, data: string|null }`

**Step 6: Add decode result display**
- Add a `<div id="decode-result">` below the QR canvas-container in the HTML
- Style it with appropriate success (green) / failure (red) colors using the existing color system
- After overlay + decode, update this div:
  - Success: "Decode: Success" with green styling and the decoded URL text
  - Failure: "Decode: Failed" with red styling
  - No pattern painted: "Decode: No pattern applied" in neutral gray

**Step 7: Wire the pipeline**
Update `generateQR()` to call the full pipeline:
1. Generate QR model (existing)
2. Render QR to our own canvas (`renderQRToCanvas`)
3. Create function pattern mask (`createFunctionPatternMask`)
4. Overlay painted pixels (`overlayPaintedPixels`) -- only if paintGrid exists and has any locked pixels
5. Decode corrupted QR (`decodeCorruptedQR`)
6. Display result

Also add a "re-overlay" trigger: when the user paints a pixel (clicks on the paint canvas), automatically re-run steps 2-6 so the QR display updates in real-time. This means after each paint click:
- Re-render the clean QR to canvas (from the stored model)
- Re-overlay with current paint state
- Re-decode and update status

Store the QR model globally so we can re-render without regenerating.

**Important notes:**
- The QR canvas cell size and paint canvas cell size may differ (QR canvas is for display, paint canvas is for interaction). The overlay maps paint grid coordinates to QR canvas pixel coordinates.
- Research flagged that jsQR does NOT expose Reed-Solomon error counts -- only binary success/failure. This is acceptable for Phase 2. Requirement QR-04 asks for "decoder error counting" but research recommended starting with binary decode. Display success/fail for now and document the gap. The roadmap success criteria says "displays decoder error count" -- we will display "Decode: Success" or "Decode: Failed" as the error metric, with a note that granular error counting is a Phase 3 enhancement.
- Ensure function pattern mask is recalculated when version changes (it depends on version).
