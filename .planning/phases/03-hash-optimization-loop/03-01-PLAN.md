  <action>
Create the optimization Web Worker as an inline JavaScript string (Blob URL pattern) inside the main application script section. This is the most complex task in the phase -- it requires extracting QR encoding from qrcodejs and bundling it with jsQR inside the worker.

**Step 1: Extract QR encoding logic for worker use**

The qrcodejs library (already inlined in index.html at line 419) has DOM-dependent rendering code but the core QR model generation is pure math. Extract the QR model generation into worker-compatible code:

- Copy the core QR encoding classes from qrcodejs: the `b` constructor (QR model), `a` (data parser), `i` (polynomial), `j` (RS blocks), `k` (bit buffer), `f` (utilities/patterns), `g` (galois field), and the `c`, `d`, `e` constants.
- The critical function is `b.prototype.make()` which calls `makeImpl()` to generate the module matrix.
- Strip all DOM rendering code (the `o`, `p`, `q` drawing classes and the `QRCode` wrapper constructor).
- Create a pure function `generateQRModel(text, version, errorCorrectionLevel)` that:
  1. Creates a new `b(version, errorCorrectionLevel)` instance
  2. Adds data via `addData(text)`
  3. Calls `make()`
  4. Returns the modules array (2D boolean matrix)
- Test approach: The worker's `generateQRModel` for a given URL+version must produce identical module matrix as `qrCodeInstance._oQRCode.modules` on the main thread.

**Step 2: Include jsQR in worker**

The jsQR library is already inlined (line 424 onward, ~10K lines). For the worker:
- Include the jsQR source inside the worker code string. Since it's a self-contained UMD module, wrap it so `jsQR` is available as a function inside the worker scope.
- The jsQR factory function uses `typeof self !== 'undefined' ? self : this` which works perfectly in Worker context.

**Step 3: Implement the search pipeline inside the worker**

The worker's core loop for each iteration:
1. Generate random hash fragment (8 chars from URL-safe charset `ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_`) using `crypto.getRandomValues()`
2. Construct full URL: `baseURL + '#' + hash`
3. Generate QR model using extracted `generateQRModel(fullURL, version, 2)` (2 = error correction H)
4. Get module count from model, compare with painted pixel grid:
   - For each painted pixel (state 1=white or 2=black) that is NOT in the function pattern mask:
     - Compare painted color with QR module dark/light state
     - Count mismatches (pixel diff)
5. Decode: render model to ImageData (1px per module, pure black/white), overlay painted pixels onto ImageData (skip function mask positions), then run jsQR on the resulting ImageData
6. Record result: { hash, pixelDiff, decodable (boolean), moduleData (compact representation for display) }
7. Update top 5 tracking: sort by decodable (true first), then pixelDiff (lower first)

**Step 4: Worker message protocol**

Messages FROM main thread TO worker:
- `{ type: 'START_OPTIMIZATION', baseURL, version, paintedPixels, functionMask, timeout, existingTopResults }`
  - `paintedPixels`: flat array of pixel states (0/1/2) from PixelGrid.cells
  - `functionMask`: flat boolean array from createFunctionPatternMask()
  - `timeout`: milliseconds
  - `existingTopResults`: array of previous top results (for accumulation across runs)
- `{ type: 'STOP_OPTIMIZATION' }`

Messages FROM worker TO main thread:
- `{ type: 'PROGRESS', attempts, elapsed, topResults }` -- sent every 100 iterations or 500ms (whichever comes first)
- `{ type: 'COMPLETE', attempts, elapsed, topResults, decodableCount }` -- sent when timeout expires or stop received

Each result in `topResults` array: `{ hash, pixelDiff, decodable, moduleData }` where `moduleData` is a flat Uint8Array of module dark/light values (1 byte per module, 0=light 1=dark, with painted pixels overlaid).

**Step 5: Wire worker creation in main thread**

Add to the main application script:
- `let optimizationWorker = null;`
- `function createOptimizationWorker()` that creates the Blob URL worker
- `function terminateOptimizationWorker()` for cleanup
- Worker onmessage handler that stores incoming progress/complete data (don't update UI yet -- Plan 02 handles that)
- A simple `startOptimization()` function that gathers current state (currentURL, currentVersion, paintGrid.cells, createFunctionPatternMask(currentVersion), timeout) and posts START_OPTIMIZATION
- A simple `stopOptimization()` function that posts STOP_OPTIMIZATION

**Important implementation notes:**
- The worker code string will be large (it includes the full QR encoding logic + jsQR library). This is expected and correct for single-file architecture.
- Use template literal (backtick string) for the worker code to allow multi-line.
- The QR encoding extraction is the hardest part. Study the qrcodejs minified source carefully. The key classes are: `b` (QR model with makeImpl, isDark, getModuleCount), `f` (pattern tables, BCH, masking), `g` (galois field), `i` (polynomial), `j` (RS blocks), `k` (bit buffer), `a` (data parser), and constants `c`, `d`, `e`, `l` (capacity table).
- For the pixel diff calculation, work at module level (not pixel level) -- compare module dark/light with painted pixel state directly. Only count painted pixels that are NOT in function mask positions.
- The `shouldStop` flag in the worker must be checked between iterations to allow clean stopping.
- For progress reporting frequency: send PROGRESS every 100 attempts OR every 500ms, whichever triggers first. This prevents flooding the main thread while ensuring responsive UI updates.
