Purpose: Longer hashes create more variation in QR data regions, giving the optimization search more candidates that may align with the painted pattern. Currently fixed at 4 chars, but QR versions can hold much more.

Output: Dynamic hash length that maximizes capacity usage while maintaining minimum length.

3. Add `calculateDynamicHashLength(url, version)` function after calculateMinimumVersion():
   ```javascript
   /**
    * Calculate optimal hash length to fill available QR capacity
    * @param {string} url - Base URL without hash
    * @param {number} version - QR version (1-10)
    * @returns {number} Hash length (at least MIN_HASH_LENGTH)
    */
   function calculateDynamicHashLength(url, version) {
     const urlBytes = new Blob([url]).size;
     const hashPrefixBytes = 1; // '#'
     const capacity = CAPACITIES_H_BYTE.find(e => e.version === version)?.capacity || 0;
     
     // Available bytes for hash = capacity - url bytes - '#'
     const availableBytes = capacity - urlBytes - hashPrefixBytes;
     
     // Return at least MIN_HASH_LENGTH, at most available bytes
     return Math.max(MIN_HASH_LENGTH, availableBytes);
   }
   ```

This function calculates how many hash characters can fit in the remaining QR capacity after the URL and '#' are accounted for.
